## 테스트 4: RabbitMQ STOMP 브로커 방당 500명 검증

**스크립트:** `tests/load/phase3/stomp-rabbitmq.js`
**시나리오:** 방 2개, VU 0 → 200 → 500 → 500(유지) → 1000 → 1000(유지) → 0 (총 3분 46초)
**목표:** SimpleBroker(200명/방 한계) → RabbitMQ로 방당 500명 달성 여부 확인
**설계:** setup()에서 유저 1,000명 사전 생성 후 VU가 재사용 (매 반복 유저 생성 제거)

### 결과 요약

| 지표 | 값 | 임계값 | 판정 |
|------|----|--------|------|
| HTTP 요청 실패율 | **0.00%** | < 1% | ✅ 통과 |
| HTTP 응답시간 p(95) | **3,639ms** | < 500ms | ❌ 초과 |
| WS 연결 에러 수 | **0** | < 20 | ✅ 통과 |
| STOMP 연결 에러 수 | **767** | < 20 | ❌ 초과 |
| WS 성공률 | **94.3%** | > 95% | ❌ 미달 (0.7%p 부족) |
| WS 세션 p(95) | **15,532ms** | < 10,000ms | ❌ 초과 |
| STOMP 수신 메시지 | **1,593,257건** | > 0 | ✅ 통과 |
| 총 이터레이션 | 14,934건 | — | — |
| 최대 동시 VU | **1,000** | — | — |

### 퍼센타일 분포

| 지표 | avg | med | p(90) | p(95) | max |
|------|-----|-----|-------|-------|-----|
| HTTP 응답시간 | 959ms | 431ms | 2,598ms | 3,639ms | 11,070ms |
| WS 세션 지속시간 | 6,108ms | 3,943ms | 15,019ms | 15,532ms | 19,247ms |

### 메시지 처리량

| 항목 | 값 |
|------|---:|
| STOMP 메시지 발신 | 69,522건 |
| STOMP 메시지 수신 (브로드캐스트) | 1,593,257건 |
| WS 메시지 발신 | 112,110건 |
| WS 메시지 수신 | 1,607,561건 |
| 수신/발신 비율 | **22.9× fan-out** |

**분석:**
- **RabbitMQ STOMP Relay 동작 확인**: 69,522건 발신 → 1,593,257건 수신 (**22.9배 fan-out**)으로 크로스 인스턴스 브로드캐스트 정상 동작
- **유저 사전 생성 효과**: MySQL INSERT 병목 제거 후 이터레이션 43% 증가 (10,464 → 14,934), 메시지 수신 3배 증가 (522K → 1,593K)
- **WS 성공률 94.3%**: 임계값 95%에 0.7%p 부족 → RabbitMQ 단일 노드의 동시 STOMP 연결 한계
- **Phase 2 SimpleBroker 대비**: Phase 2에서는 200명/방까지만 안정이었으나, RabbitMQ로 전환 후 **500명/방 구간까지 메시지 전파 성공**
- **병목 지점**: 로컬 Docker RabbitMQ 단일 노드에서 1,000 동시 STOMP 연결 시 연결 큐잉 → 15초 타임아웃 초과
- **개선 방향**: RabbitMQ 클러스터링 또는 연결 풀링으로 고동시성 해소 가능

## 테스트 5: 다중 인스턴스 로드밸런싱 검증

**스크립트:** `tests/load/phase3/multi-instance-lb.js`
**시나리오:** 방 10개, VU 0 → 200 → 500 → 1000 → 1500 → 1500(유지) → 500 → 0 (총 6분 23초)
**목표:** Nginx LB를 통한 chat-server-1/2 분산 처리, 1,500 VU 동시 접속
**설계:** setup()에서 유저 1,500명 + 방 10개 사전 생성 후 VU가 재사용

### 결과 요약

| 지표 | 값 | 임계값 | 판정 |
|------|----|--------|------|
| HTTP 요청 실패율 | **48.13%** | < 5% | ❌ 초과 |
| API 성공률 | **51.2%** | > 95% | ❌ 미달 |
| 연결 오류 수 | **13,673** | < 100 | ❌ 초과 |
| 입장 응답시간 p(95) | **8,160ms** | < 3,000ms | ❌ 초과 |
| 메시지 조회 p(95) | **626ms** | < 500ms | ❌ 초과 |
| HTTP 전체 p(95) | **8,245ms** | < 5,000ms | ❌ 초과 |
| 총 이터레이션 | 28,243건 | — | — |
| 총 HTTP 요청 수 | 142,781건 | — | — |
| 최대 동시 VU | **1,500** | — | — |

### 단계별 성공률 분석

| 검증 항목 | 성공 | 실패 | 성공률 |
|-----------|-----:|-----:|-------:|
| 채팅방 입장 | 14,585 | 13,673 | **51.6%** |
| 메시지 조회 | 14,224 | 14,034 | **50.3%** |
| 채팅방 목록 | 14,581 | 13,673 | **51.6%** |

### 퍼센타일 분포

| 지표 | avg | med | p(90) | p(95) | max |
|------|-----|-----|-------|-------|-----|
| HTTP 전체 응답시간 | 1,461ms | 9.35ms | 5,534ms | 8,245ms | 58,027ms |
| 입장 응답시간 | 1,565ms | 8.40ms | 5,640ms | 8,160ms | 20,317ms |
| 메시지 조회 (api-server) | 159ms | 36.23ms | 430ms | **626ms** | 3,133ms |

**분석:**
- **유저 사전 생성 후 병목 전환**: 이전 테스트에서 MySQL INSERT(유저 생성)이 95% 실패의 원인이었으나, 사전 생성 방식으로 전환 후 MySQL 병목 완전 해소
- **현재 병목 — TCP 소켓 한계**: 1,500 VU에서 `connection forcibly closed` (TCP reset) 발생 → 로컬 Windows Docker Desktop의 TCP 소켓 수 제한
- **성공한 요청은 정상 동작**: 입장/목록 조회 약 14,500건이 성공적으로 처리되어 Nginx LB + 2인스턴스 분산 자체는 안정
- **api-server 메시지 조회**: p(95) 626ms로 1,500 VU 부하에서는 다소 증가하나, med 36ms로 대부분의 요청은 빠르게 처리
- **실질 한계**: 로컬 Docker 환경에서 chat-server 2대 기준 약 **1,000 VU**까지가 REST API 안정 한계 (TCP 소켓 제약)
