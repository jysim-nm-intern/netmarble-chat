## SDD란 무엇인가

**Spec-Driven Development(스펙 주도 개발)**은 코드 작성 이전에 시스템이 "무엇을 해야 하는가(What)"를 정밀하게 기술한 **명세(Specification)**를 먼저 확정하고, 이 명세를 유일한 진실의 원천(Single Source of Truth)으로 삼아 구현과 검증을 진행하는 개발 방법론입니다.

```
[전통적 방식]
요구사항 → 개발 → 테스트 → 문서화 (사후)

[SDD 방식]
요구사항 → 명세 작성 → AI 코드 생성 → 명세 기반 검증 → 배포
              ↑_________________________________↑
                   (명세가 코드와 테스트를 모두 주도)
```

### SDD의 핵심 원칙

| 원칙 | 설명 |
|------|------|
| **명세 우선** | 코드보다 명세가 먼저 존재한다. 코드는 명세를 충족시키기 위해 작성된다. |
| **계약 기반 설계** | 각 컴포넌트는 명확한 입력/출력 계약(Contract)으로 정의된다. |
| **인수 조건** | 모든 기능 명세는 자동화 가능한 인수 조건(Acceptance Criteria)을 포함한다. |
| **AI 최적화** | 명세는 AI가 명확한 코드를 생성할 수 있도록 충분한 컨텍스트를 제공한다. |
| **기술 독립성** | 명세는 특정 구현 기술에 종속되지 않는다. |

### AI 기반 개발에서의 SDD 효용

- **코드 일관성**: 명세를 기준으로 생성된 코드가 프로젝트 전체에서 일관된 패턴을 따릅니다.
- **재작업 최소화**: "원하는 것"이 명세로 고정되어 AI가 방향을 잃지 않습니다.
- **검증 자동화**: 명세의 인수 조건이 곧 테스트 코드의 기준이 됩니다.
- **팀 간 소통**: 개발자와 AI 사이의 공통 언어 역할을 합니다.
