## 3. 해결된 병목

### 3.1 N+1 쿼리 문제
- **증상:** 채팅방 목록 조회 시 302개 쿼리 발생
- **원인:** `@ManyToOne` LAZY 로딩 + 반복 호출
- **해결:** DTO Projection (`SELECT new`) + `JOIN FETCH`
- **결과:** 302 → 3~4 쿼리/요청

### 3.2 메시지 저장/조회 성능
- **증상:** MySQL offset-based 페이징 → 데이터 증가 시 성능 저하
- **원인:** `OFFSET N` 스캔 비용이 N에 비례
- **해결:** MongoDB cursor-based 페이징 (`{ _id: { $lt: cursor } }`)
- **결과:** P99 7.04ms (5,000건 기준, 200 VU 동시)

### 3.3 읽음 상태 동시성
- **증상:** MySQL UPDATE 경합 → 데드락 가능성
- **원인:** 동시 읽음 처리 시 행 레벨 락 경합
- **해결:** Redis Lua Script (SADD + INCR 원자적 실행)
- **결과:** 100명 동시 읽음 처리 안정

### 3.4 세션 기반 인증
- **증상:** HttpSession → 수평 확장 불가 (Sticky Session 필요)
- **원인:** 서버 메모리에 세션 저장
- **해결:** JWT + Redis 블랙리스트 (Stateless)
- **결과:** 2서버 라운드 로빈 10/10 통과
