## 3. 해결된 병목

### 3.1 SimpleBroker 방당 200명 한계

- **증상:** 200명 초과 시 STOMP 연결 오류 급증, 메시지 유실
- **원인:** 인메모리 SimpleBroker의 단일 스레드 브로드캐스트 한계
- **해결:** RabbitMQ STOMP Relay 전환 (STOMP 플러그인 활성화)
- **결과:** 방당 500명 구간까지 메시지 전파 성공, 22.9× fan-out 확인

### 3.2 크로스 인스턴스 메시지 동기화

- **증상:** chat-server 복제 시 인스턴스 간 STOMP 메시지 전달 불가
- **원인:** SimpleBroker는 JVM 로컬 메모리에서만 동작
- **해결:** RabbitMQ를 중앙 브로커로 사용, 모든 인스턴스가 같은 exchange/queue 구독
- **결과:** chat-server 2대에서 69,522건 발신 → 1,593,257건 수신 (크로스 인스턴스)

### 3.3 이미지 Base64 DB 저장

- **증상:** 이미지 포함 메시지의 DB 레코드 크기 비대, 전송 성능 저하
- **원인:** 이미지를 Base64 문자열로 MongoDB에 직접 저장
- **해결:** 로컬 파일 시스템 저장 + Nginx 정적 서빙, DDD 인터페이스 설계
- **결과:** 업로드 p(95) 74ms, 100% 성공률 (100 VU)

### 3.4 수평 확장 기반 부재

- **증상:** chat-server 단일 인스턴스로 용량 한계
- **원인:** 로드밸런서 없음, 세션/브로커 공유 미지원
- **해결:** Nginx 리버스 프록시 (WebSocket 업그레이드, 라운드 로빈) + Docker Compose 복제
- **결과:** chat-server 2대 안정 동작, 인스턴스 추가로 선형 확장 가능

## 6. 향후 방향: 타 언어 마이그레이션 테스트

Phase 1~3을 통해 Spring Boot(Java) 기반으로 실시간 채팅 서버의 기능 완성도와 확장성을 검증했다. 다음 단계로 **동일한 기능 스펙과 부하 테스트 시나리오**를 다른 언어/런타임으로 재구현하여 성능·생산성·운영성을 비교 검증한다.

### 6.1 마이그레이션 테스트 목적

- 동일 스펙(SDD) 기준으로 **언어별 처리량, 응답 시간, 메모리 사용량** 정량 비교
- WebSocket/STOMP 실시간 처리에서 **이벤트 루프 vs 스레드풀** 모델의 실질 차이 측정
- 기존 k6 부하 테스트 스크립트(Phase 1~3)를 그대로 재사용하여 **공정한 벤치마크** 확보
- 팀 내 기술 스택 선정 근거 마련

### 6.2 후보 언어/프레임워크

| 후보 | WebSocket 지원 | STOMP 지원 | 특징 |
|------|:-:|:-:|------|
| **Go** (Gorilla/nhooyr + custom STOMP) | O | 직접 구현 | 경량 고루틴, 낮은 메모리, 빠른 빌드 |
| **Rust** (Actix-web / Axum + Tokio) | O | 직접 구현 | 제로코스트 추상화, 최소 메모리, 높은 처리량 |
| **Node.js** (NestJS + Socket.IO/ws) | O | O (stompit) | 이벤트 루프, 프론트엔드와 언어 통일 |
| **Kotlin** (Ktor / Spring WebFlux) | O | O | 코루틴 기반, Java 생태계 호환 |

### 6.3 비교 기준

| 카테고리 | 측정 항목 |
|----------|-----------|
| **처리량** | 동시 1,000 VU 기준 초당 메시지 처리량, fan-out 배율 |
| **응답 시간** | REST API p(95)/p(99), WebSocket 세션 지속시간 |
| **리소스** | Docker 컨테이너 CPU 사용률, 메모리(RSS), GC 빈도 |
| **안정성** | 1시간 장기 부하(soak test) 시 메모리 누수, 에러율 추이 |
| **개발 생산성** | 구현 소요 시간, 코드 라인 수, 의존성 수 |
| **운영성** | Docker 이미지 크기, 기동 시간, 프로파일링 도구 성숙도 |

### 6.4 테스트 전략

1. **기능 스펙 고정**: 현재 SDD 문서(SPEC-001~007)의 기능 요구사항을 그대로 준수
2. **인프라 동일**: MySQL + MongoDB + Redis + RabbitMQ + Nginx 구성 유지
3. **부하 테스트 재사용**: `tests/load/` 디렉토리의 k6 스크립트를 그대로 실행
4. **단계적 진행**: 핵심 기능(유저/채팅방/메시지 CRUD + STOMP) 먼저 구현, 이후 이미지/읽음 처리 확장
5. **결과 문서화**: `docs/migration-benchmark.md`에 언어별 벤치마크 결과 정리

### 6.5 기대 효과

- Spring Boot 대비 **Go/Rust의 메모리 효율** (JVM 2GB vs 수십~수백 MB 예상)
- **컨테이너 기동 시간** 단축 (JVM warm-up 제거)
- WebSocket 동시 연결 처리에서 **고루틴/async 모델의 이점** 정량화
- 반대로 **Spring 생태계의 생산성 우위** (STOMP 내장, JPA, Security 등) 수치화

## 7. 결론

Phase 3에서는 실시간 채팅 서버의 **확장성 기반**을 완성했다.

| 목표 | 달성 여부 |
|------|:---------:|
| RabbitMQ STOMP Relay 전환 | ✅ 방당 500명, 22.9× fan-out |
| chat-server 수평 확장 | ✅ Nginx LB → 2인스턴스, 크로스 인스턴스 메시지 |
| 이미지 저장소 분리 | ✅ 로컬 파일 + Nginx 서빙, p(95) 74ms |
| 부하 테스트 | ✅ k6 3종 시나리오, Phase 1~3 비교 리포트 |
| 1,000 VU 안정성 | ✅ REST 실패율 0%, API 성공률 100% |
| 1,500 VU 확장 | ⚠️ 로컬 Docker TCP 한계, 클라우드 재검증 필요 |

Phase 1(단일 서버 400명) → Phase 2(2서버 1,000명) → Phase 3(N서버 확장 가능 + RabbitMQ)으로 이어진 아키텍처 진화를 통해 **동일 SDD 스펙 위에서 점진적으로 확장성을 확보**했다. 이 동일한 스펙과 테스트 인프라를 활용하여, 향후 타 언어 마이그레이션 테스트를 통해 기술 스택 선정의 정량적 근거를 확보할 수 있다.
