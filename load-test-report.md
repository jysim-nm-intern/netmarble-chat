## 동시 채팅 수용 인원 분석

### 방 1개 기준 (단일 채팅방 수용 한계)

단일 방 스트레스 테스트에서 VU를 단계적으로 증가시켜 측정한 결과입니다.

| 구간 | 동시 인원 | HTTP 상태 | WS/STOMP 상태 | 판정 |
|------|-----------|-----------|---------------|------|
| 정상 구간 | **~200명** | avg 낮음, 실패 0% | STOMP 오류 0건, 성공률 100% | ✅ 안정 |
| 경고 구간 | **200~350명** | 응답 지연 증가 | STOMP 오류 산발 시작 | ⚠️ 주의 |
| 임계 구간 | **350명~** | p(90) 3598ms | STOMP 오류 급증, 성공률 하락 | ❌ 위험 |
| 한계 구간 | **500명** | p(95) 5086ms | STOMP 오류 860건, 성공률 83.9% | ❌ 한계 초과 |

> **결론: 방 1개 기준 안전 동시 인원 = 약 200명**
>
> - 200명까지는 STOMP 오류 0건, WS 성공률 100% (websocket-stomp 테스트에서 방당 100명 검증)
> - 350명부터 SimpleBroker 구독 목록 탐색 병목으로 연결 타임아웃 급증
> - 200명 초과 시 성능이 선형이 아닌 급격(non-linear)하게 저하됨

**단일 방 퍼센타일 응답시간 (전체 VU 구간 통합):**

| 지표 | p(50) | p(90) | p(95) | max |
|------|-------|-------|-------|-----|
| HTTP 응답시간 | ~avg 1130ms | 3598ms | 5086ms | 18300ms |
| WS 세션 시간 | ~avg 7879ms | 20411ms | 21313ms | 28896ms |

### 전체 서비스 기준 (총 동시 채팅 수용 한계)

다중 방 분산 테스트에서 검증된 수치입니다.

| 구성 | 총 동시 VU | HTTP 상태 | WS/STOMP 상태 | 판정 |
|------|------------|-----------|---------------|------|
| 8방 × 50명 | **400명** | p(95) 1051ms, 실패 0% | 오류 0건, 성공률 100% | ✅ 검증됨 |
| 방당 200명 한계 적용 시 | **400~800명 추정** | REST 병목 없음 | 방 분산 시 선형 확장 가능 | 📊 추정치 |

> **결론: 현재 서버 스펙 기준 전체 서비스 안전 동시 인원 = 최소 400명 (실측), 최대 400~800명 (추정)**
>
> - **실측 확인된 수치: 400명** (8방 × 50명, 100% 성공)
> - **REST API 병목 없음**: 500 VU에서도 HTTP 실패율 0% → REST 계층은 현재 설정으로 충분
> - **병목은 단일 방 집중**: 방을 분산하면 400 VU 이상도 선형 확장 가능
> - 이론상 방당 200명 × (Tomcat 스레드 400 / 방당 평균 사용 스레드) = 방 수에 따라 확장

**방 구성별 이론 수용 인원 (방당 200명 한계 적용):**

| 방 수 | 방당 인원 | 총 동시 인원 | 근거 |
|-------|-----------|------------|------|
| 2방 | 100명 | 200명 | websocket-stomp 테스트 실측 (100%) |
| 8방 | 50명 | 400명 | multi-room 테스트 실측 (100%) |
| 8방 | 200명 | **1,600명** | 방당 한계 최대 적용 시 이론치 |
| N방 | 200명 | **N × 200명** | 방 분산 선형 확장 예상 (미검증) |

> ⚠️ 방당 200명 × 다수 방 시나리오(ex. 1,600명 이상)는 이번 테스트에서 검증되지 않았습니다.
> 브로커(SimpleBroker) 자체의 전체 구독 수 한계가 별도로 존재할 수 있습니다.

### 확인된 한계
- **단일 방 동시 접속 임계점: 약 350 VU**
  SimpleBroker는 단일 채널 구독 목록을 순차 탐색하므로 동일 방에 대규모 동시 접속 시 성능 저하

- **REST API는 500 VU에서도 실패율 0%로 매우 안정적**
  HikariCP 50 커넥션 + Tomcat 400 스레드 설정이 유효하게 동작
